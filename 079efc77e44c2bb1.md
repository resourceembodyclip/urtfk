# LinkWiki Core

LinkWiki Core 是一个面向技术文档维护者、知识库管理员和开源项目贡献者的结构化外链资源汇总与导航系统。该项目并非简单的书签集合，而是一套带有元数据标注、批次管理和溯源追踪的轻量级资源索引框架。它解决了技术团队在协作过程中出现的文档引用散乱、外部链接失效难以追溯、以及资源发现依赖口头传播等典型问题。LinkWiki Core 定位于中大型技术文档库的底层链接治理层，可作为 MkDocs、VuePress 或 Docusaurus 站点的数据源插件使用。

## 功能概览

**批次化资源导入** 支持按批次（当前为第 27/56 批）对大量 URL 进行结构化录入，自动生成资源清单与统计摘要。

**原始 URL 严格保真** 系统对用户输入的每一枚链接均做原样存储，不自动补全协议头、不修正大小写、不添加尾部斜杠，确保与上游数据源完全一致。

**多维度分类视图** 资源可按来源域名、内容主题、文档类型等维度自动归类，便于不同角色的使用者快速定位所需外链。

**元数据扩展字段** 每条资源记录支持附加说明标签、收录时间、校验状态（有效/失效）及关联的文档内部引用计数。

**轻量级部署架构** 基于静态 Markdown 文件驱动，无需独立数据库，可与 Git 工作流无缝集成，实现链接变更的版本化追踪。

**命令行交互工具** 提供 CLI 接口用于批量链接健康检查、重复项检测以及 Markdown 索引表的自动生成。

## 应用场景

技术文档站的链接治理
对于维护数百页 API 文档或用户手册的团队，LinkWiki Core 可作为独立的链接登记簿，在文档写作过程中统一管理所有外部参考链接。当第三方文档发生迁移时，管理员可通过资源索引快速定位受影响页面，避免文档中出现大量死链。

开源项目的资源汇总页
开源项目通常需要在 README 或 Wiki 中列出依赖项目、学习资料或社区扩展。LinkWiki Core 的批次管理能力允许维护者按版本发布周期整理资源列表，确保每次发布对应的外部参考快照清晰可查。

知识库的引用溯源
企业内部知识库或技术博客常需引用行业标准、 RFC 文档或商业产品页面。使用 LinkWiki Core 记录每一枚外链的来源批次和收录时间，可在版权审计或内容合规审查时提供完整的引用链条。

多仓库共享资源池
在微服务架构或多项目组织中，不同代码仓库的文档可能引用相同的公共资源。LinkWiki Core 可作为中心化的资源索引子模块，通过 Git Submodule 或子树合并方式被多个仓库复用，避免资源链接在各仓库间不一致。

## 快速开始

以下命令演示如何在本地环境中获取并运行 LinkWiki Core 的基础资源索引生成功能。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkwiki-core.git
cd linkwiki-core

# 安装依赖（基于 Python 3.9+）
pip install -r requirements.txt

# 执行资源索引生成，输出 markdown 格式的资源列表
python cli.py build --batch 27 --output ./docs/resources.md
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 或更高 | 核心运行环境，用于 CLI 工具和元数据处理 |
| pip | 21.0 或更高 | Python 包管理器，用于安装项目依赖 |
| Git | 2.25 或更高 | 版本控制，用于克隆仓库和提交资源变更 |
| Markdown 解析库 | Python-Markdown 3.3.6 | 用于将资源数据渲染为结构化 Markdown 表格 |
| 网络连接 | 出站 HTTPS 可达 | 执行链接健康检查时需访问外部域名 |
| 操作系统 | Linux / macOS / Windows WSL2 | 开发与生产环境均支持 Unix-like 系统 |

## 项目结构

```
linkwiki-core/
├── cli.py                     # 命令行入口，注册 build/check/export 子命令
├── requirements.txt           # Python 依赖声明，包含 Markdown 解析与网络请求库
├── config/
│   ├── default.yaml           # 默认配置项：批次大小、检查超时、输出目录
│   └── schema.json            # 资源配置文件的 JSON Schema 校验定义
├── core/
│   ├── __init__.py
│   ├── loader.py              # 从纯文本或 CSV 导入 URL 列表，做基本格式清洗
│   ├── registry.py            # 资源注册表核心类，管理批次与单个条目的 CRUD
│   ├── validator.py           # URL 语法校验与重复检测逻辑
│   └── health.py              # 并发 HTTP 请求检查链接可达性，记录状态码
├── renderers/
│   ├── markdown.py            # 将资源数据渲染为带章节标题的 Markdown 列表
│   ├── html.py                # 生成用于静态站点侧边栏的 HTML 导航片段
│   └── json.py                # 导出为 JSON 格式供其他工具链消费
├── docs/                      # 项目自身文档源文件
│   ├── user-guide/
│   ├── admin/
│   ├── dev/
│   └── design/
├── tests/                     # 单元测试，覆盖加载器、注册表和健康检查模块
│   ├── test_loader.py
│   ├── test_registry.py
│   └── test_health.py
└── samples/                   # 示例数据，包含演示批次和资源列表模板
    ├── batch_27_sample.txt
    └── expected_output.md
```

## 贡献指南

提交问题报告
在 GitHub Issues 中提交 bug 或功能请求时，请附带 LinkWiki Core 的版本号、Python 环境信息以及可复现问题的最小输入数据（例如一段 URL 列表）。

添加新批次数据
若需贡献新的资源批次，请将 URL 列表整理为纯文本文件，每行一条，放置于 samples/ 目录下，并附带简要的批次说明（来源、收录范围）。提交前请运行 cli.py validate 确保无格式错误。

改进渲染器
渲染器模块设计为可插拔架构。如需新增输出格式（如 CSV 或 Atom Feed），请继承 base.Renderer 基类，实现 render 方法，并在 cli.py 中注册子命令。提交时需包含相应的单元测试。

完善文档
文档源位于 docs/ 目录，采用 Markdown 编写。请确保新增或修改的内容与代码实际行为一致。涉及命令行参数的更新需同步更新 docs/dev/cli-commands.md。

本地测试流程
提交 Pull Request 前，请确保在本地通过全部测试（pytest tests/），并执行一次完整构建（python cli.py build --batch all）验证无异常输出。

## 常见问题

Q: 项目是否会自动将裸域名补全为 https:// 格式？
A: 不会。LinkWiki Core 的设计原则是绝对保真存储。用户输入的裸域名（如 example.com）会原样保存在资源记录中，不会自动添加协议前缀或 www 子域名。这一设计确保与上游数据源的原始格式保持完全一致，避免因自动改写导致的引用偏差。

Q: 如何检查一批资源中是否存在重复链接？
A: 使用 CLI 的检查功能：python cli.py check --batch 27 --duplicate。该命令会对当前批次内的 URL 做归一化比较（忽略尾部斜杠差异），输出重复项列表。注意，归一化仅用于检测，不影响实际存储的原始值。

Q: 链接健康检查是否会访问所有外链？
A: 默认情况下，健康检查会发起实际的 HTTP HEAD 请求以验证链接可达性。对于数量较大的批次，建议使用 --limit 参数分批次执行，或配合 --timeout 设置较长的超时阈值，避免因网络波动产生误报。
