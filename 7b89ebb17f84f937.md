# WapWiki Resource Aggregator

WapWiki Resource Aggregator 是一个面向技术研究、知识管理及信息检索场景的轻量级外链汇总与导航系统。该项目以结构化方式收录并索引来自 wap.baike.zr54.cn 的大量百科类条目链接，为开发者、数据分析师、内容研究者提供可快速检索、可批量处理的半结构化知识索引。

本项目不直接提供内容存储，而是通过 URL 规范整理、元信息标注与分类框架，帮助用户在海量百科链接中快速定位高价值条目，减少人工检索成本，适用于知识图谱构建、语料采集、主题聚类等二次开发任务。项目定位为技术资源索引工具，面向具备基础脚本能力（Python / Shell）的中高级开发者，同时也适用于需要系统化查阅百科条目的日常研究场景。

## 功能概览

**批量链接导入**：支持从纯文本文件或 CSV 中导入 URL 列表，自动去重并校验 HTTP 可访问性，生成标准化索引库。

**分类标签生成**：基于 URL 路径中的数字 ID 区间与预设规则库，为每一条链接自动分配初步分类标签，涵盖科技、历史、文化、地理、生物等多个知识领域。

**检索过滤接口**：提供命令行查询工具，支持按关键词、ID 范围、分类标签等多维度过滤输出结果，便于定向获取链接子集。

**元数据缓存机制**：对已访问的百科条目进行标题与摘要缓存，减少重复网络请求，提升批量处理效率。

**导出格式多样化**：支持将索引结果导出为 JSON、CSV、Markdown 表格三种格式，兼容主流数据处理工具与笔记软件。

**定期更新检测**：内置简易变更检测模块，可定期检查链接有效性及页面标题变动，辅助维护索引新鲜度。

**低依赖部署**：仅依赖 Python 标准库与 requests、beautifulsoup4 两个外部包，安装与运行成本极低。

## 应用场景

知识图谱初步构建：研究者可通过本项目的链接索引，快速获取某一主题下的大量百科条目 URL，作为知识图谱实体抽取的种子链接集，配合爬虫框架进行深度信息提取。

语料采集与文本分析：自然语言处理从业者可利用导出功能批量获取百科条目链接，供后续正文抓取与语料清洗使用，适用于主题模型训练或词向量评估。

百科条目变更追踪：内容运营人员可借助更新检测模块，定时监控特定 ID 段条目的标题或摘要变化，及时捕获内容更新动态。

离线阅读准备：在网络受限环境下，用户可先通过本项目筛选所需条目列表，再统一安排离线抓取与归档，提高阅读效率。

技术文档内链补充：开源项目文档编写者可使用本索引作为外部参考资料池，在技术文档中快速引用相关概念解释条目。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，Python 3.8 及以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/example/wapwiki-resource-aggregator.git
cd wapwiki-resource-aggregator

# 安装依赖
pip install -r requirements.txt

# 运行初始化导入任务（示例：导入默认链接列表）
python scripts/import_urls.py --input data/default_urls.txt --output index.db
```

完成上述三步后，项目会在当前目录生成 index.db 索引文件，并输出导入统计日志。随后可使用 `query_cli.py` 进行检索。

```bash
python scripts/query_cli.py --tag 科技 --limit 10
```

## 安装要求

|------|------|------|
| Python 3.8+ | 是 | 核心运行环境，低于此版本将导致类型注解解析异常 |
| requests 2.28+ | 是 | 用于发起 HTTP 请求，检测链接可用性及获取页面标题 |

| git 2.25+ | 是 | 仅用于克隆仓库，运行阶段无需 |

## 贡献指南

欢迎各类形式的贡献，包括但不限于新增分类规则、优化解析性能、完善文档与修复缺陷。请遵循以下步骤参与本项目。

首先，在 GitHub 上 fork 本仓库，并将你的 fork 克隆至本地。创建新的功能分支，分支命名建议采用 `feature/功能描述` 或 `fix/问题简述` 格式。

其次，针对你的改动编写或更新对应的单元测试，确保测试覆盖率达到 80% 以上。所有测试用例需通过 `pytest tests/` 命令验证无误。

第三，遵循项目现有的代码风格，Python 代码需符合 PEP 8 规范，提交前使用 `black` 与 `flake8` 进行格式化与静态检查。

第四，提交 pull request 至主仓库的 main 分支，并在 PR 描述中清晰说明改动目的、实现方式及影响范围。若涉及新增依赖，请在 requirements.txt 中注明版本约束。

最后，重大功能变更或架构调整需先提交设计讨论议题（issue），经维护者确认后再进行开发，避免无效劳动。

## 常见问题

**问：导入大量链接时出现超时或连接错误，如何处理？**

答：默认单次请求超时设置为 10 秒，重试次数为 3 次。若网络环境不稳定，可在 `src/parser/html_fetcher.py` 中调整 `REQUEST_TIMEOUT` 与 `MAX_RETRIES` 变量。建议使用稳定的代理或在内网环境中执行批量导入。同时可启用 `--skip-verify` 模式仅建立索引而不实际访问页面，随后再单独执行更新检测。

**问：分类标签的规则如何自定义？**

答：分类规则位于 `src/classifier/rule_engine.py` 中的 `RULE_DICT` 字典。该字典以 ID 区间或关键词模式为键，以标签列表为值。用户可直接修改该字典，或通过外部 JSON 配置文件加载自定义规则。修改后需重新运行导入脚本以应用新规则。

**问：导出 Markdown 表格时链接过长导致排版混乱，有无解决方案？**

答：导出模块默认会对 URL 进行截断显示，但不会影响实际链接内容。若需完整显示，建议使用 CSV 或 JSON 格式导出，再通过其他工具渲染。若必须使用 Markdown，可在 `export_formatter.py` 中调整 `MAX_URL_DISPLAY_LENGTH` 参数控制显示长度。
