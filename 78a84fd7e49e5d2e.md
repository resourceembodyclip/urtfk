# WapWiki Resource Aggregator

WapWiki Resource Aggregator 是一个面向移动端知识库与百科类内容的外链聚合与导航系统。该项目定位于技术研究者、内容运营人员以及自建百科站点的开发者，提供结构化的外链资源收录、分类索引与快速检索能力。

项目解决的核心问题在于：移动端百科类站点普遍存在内容分散、外链混乱、资源不可追溯等问题。WapWiki Resource Aggregator 通过统一的收录规范与分类体系，将分散的百科条目外链进行系统化整理，使得使用者能够快速定位特定主题下的高质量参考链接，同时为自建百科站提供可扩展的外链数据模板。

本项目属于第 34/56 批资源收录批次，当前批次共收录 180 个外链资源条目，涵盖技术百科、知识问答、数据参考等多个子类。项目本身不存储任何侵权内容，仅提供外链导航服务，所有资源版权归原始站点所有。

## 功能概览

**批量外链收录**：支持按照预设分类批量导入外链资源，自动解析 URL 结构并提取条目元信息，收录效率较手动整理提升 80% 以上。

**分类索引体系**：基于主题域与内容类型构建二级分类索引，用户可按技术、人文、自然、社会等大类浏览资源，亦可通过细分子类进行精确定位。

**全文检索支持**：集成轻量级全文检索引擎，支持对资源标题、摘要、关键词以及 URL 路径进行模糊匹配检索，检索响应时间低于 200 毫秒。

**资源状态监控**：定时检测已收录外链的可访问性，自动标记失效链接并生成异常报告，保证资源列表的持续可用性与准确性。

**移动端适配界面**：专为移动设备优化的响应式前端界面，在各类屏幕尺寸下均可获得良好的浏览与检索体验，支持触屏手势操作。

**数据导入与导出**：提供 JSON 与 CSV 两种格式的数据导入导出接口，便于与其他系统进行数据交换或备份迁移。

**访问统计分析**：内置轻量级访问日志模块，记录资源点击频次与来源分布，为资源热度评估与内容优化提供数据支撑。

**自定义标签系统**：支持用户为资源条目添加自定义标签，实现多维度分类与个性化资源组织，标签数据支持云端同步。

## 应用场景

百科站点内容运营：百科类站点编辑人员可利用本系统批量管理外部参考链接，在编写词条时快速检索并引用权威外链资源，提升内容生产的效率与质量。

技术文档聚合：技术团队可将项目开发过程中参考的各类文档、教程、API 参考链接统一收录至本系统，形成团队共享的技术知识外链库，降低信息孤岛风险。

学术研究资料整理：高校师生与科研人员在进行文献综述或课题研究时，可将散落在各处的网络资料通过本系统进行结构化整理，便于后续查阅与引用。

个人知识管理：知识工作者与终身学习者可将日常阅读中发现的优质网络资源持续录入本系统，构建个人专属的知识外链网络，避免信息碎片化。

数据采集管道辅助：数据采集工程师可将本系统作为数据源管理工具，统一维护各类数据接口地址与参考文档链接，配合自动化脚本实现数据采集链路的快速配置。

## 快速开始

以下步骤将指导您在本地环境中快速部署并运行 WapWiki Resource Aggregator。

```bash
# 克隆项目仓库
git clone https://github.com/wapwiki/resource-aggregator.git

# 进入项目目录
cd resource-aggregator

# 安装项目依赖
npm install

# 配置环境变量
cp .env.example .env
# 请根据实际环境修改 .env 中的数据库连接与端口配置

# 执行数据库迁移
npm run migrate

# 导入当前批次资源数据（第 34/56 批）
npm run import -- --batch=34

# 启动开发服务器
npm run dev

# 生产环境启动
npm run build
npm start
```

访问 http://localhost:3000 即可进入系统主界面。默认管理员账号为 admin，密码请在首次启动后通过控制台输出的临时密码进行登录并修改。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | >= 18.0.0 | 项目运行时环境，推荐使用 LTS 版本 |
| npm | >= 9.0.0 | Node.js 包管理器，用于依赖安装与脚本执行 |
| PostgreSQL | >= 14.0 | 主数据库，存储资源条目、分类索引与用户数据 |
| Redis | >= 6.0 | 缓存与会话存储，用于提升检索性能与分布式部署 |
| Nginx | >= 1.20 | 反向代理服务器，生产环境推荐用于静态资源分发与负载均衡 |
| PM2 | >= 5.0 | Node.js 进程管理工具，生产环境用于守护进程与自动重启 |

建议生产环境至少配备 2 核 CPU 与 4GB 内存，存储空间依据资源条目数量而定，每万条资源约需 50MB 数据库存储空间。操作系统推荐使用 Ubuntu 20.04 LTS 或 CentOS 7 以上版本。

## 项目结构

```
resource-aggregator/
├── src/                                # 源代码主目录
│   ├── controllers/                    # 控制器层，处理HTTP请求与响应
│   │   ├── resourceController.js       # 资源条目CRUD操作控制器
│   │   ├── categoryController.js       # 分类管理控制器
│   │   └── searchController.js         # 全文检索控制器
│   ├── models/                         # 数据模型层，定义数据库Schema与业务实体
│   │   ├── Resource.js                 # 资源条目模型（标题、URL、摘要、分类、标签等）
│   │   ├── Category.js                 # 分类模型（名称、父级ID、排序权重等）
│   │   └── AccessLog.js                # 访问日志模型（资源ID、时间戳、来源IP等）
```

第四步，向主仓库的 `develop` 分支提交 Pull Request。PR 描述中请清晰说明变更内容、影响范围以及测试情况，并关联相关的 issue 编号。项目维护者将在 3 个工作日内进行 Code Review 并给出反馈。

## 常见问题

Q: 导入资源数据时提示 URL 格式验证失败，如何解决？

A: 系统对收录的 URL 进行严格格式校验，要求必须以 http:// 或 https:// 开头。如遇验证失败，请检查原始 URL 是否包含非法字符或缺失协议头。对于裸域名格式，系统不会自动补充前缀，需要手动修正后重新导入。您也可以在导入前通过 `scripts/validate-urls.js` 脚本对 URL 列表进行预检。

Q: 资源健康检查报告显示大量链接不可访问，应如何处理？

A: 健康检查服务默认每 24 小时执行一次，检测超时时间为 5 秒。如果链接因网络抖动或临时维护导致不可访问，系统会在下一轮检查中自动恢复状态。对于持续不可达的链接，建议人工核实后手动标记为失效或更新为新地址。您也可以通过管理后台的「资源编辑」功能批量修改失效链接的目标地址。

Q: 如何将本系统与现有的内容管理平台（如 WordPress、Drupal）集成？

A: 项目提供完整的 RESTful API，支持资源条目的增删改查操作。您可以通过 API 将本系统作为外链数据源嵌入现有平台。具体集成方式请参考 docs/api-reference.md 文档中的认证与调用示例。对于 WordPress 用户，社区提供了非官方的插件适配层，可在 WordPress 插件目录中搜索「WapWiki Integration」获取。
