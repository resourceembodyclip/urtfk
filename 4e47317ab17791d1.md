# WapWiki Resource Aggregator

WapWiki Resource Aggregator 是一个面向移动端知识库索引与技术资料外链汇总的开源项目。该项目系统性地收集、分类并维护来自 wap.baike.zr54.cn 的高质量百科条目与技术文档链接，为开发者、研究人员及内容创作者提供结构化、可检索的外部知识引用体系。

项目定位为技术资源导航工具，解决用户在分散网络环境中查找可靠技术参考资料时的效率问题。通过统一的条目标识符体系与稳定的 URL 模式，WapWiki Resource Aggregator 将碎片化的移动端百科内容转化为可复用、可审计的知识资产，适用于文档系统集成、自动化引用校验以及人工知识图谱构建等场景。

## 功能概览

**批量链接采集** 支持对指定域名下的百科条目进行大规模自动化采集，本批次收录 180 条有效链接，覆盖多学科领域。

**条目元数据解析** 从每条 URL 中提取条目唯一标识符（即 `details/` 后的数字部分），便于后续数据库索引与去重处理。

**分类标签体系** 根据 URL 模式与条目编号区间，自动归入预设的技术栈分类（如后端开发、前端工程、数据科学、运维监控等）。

**变更检测与通知** 周期性检查已收录链接的可访问性及内容更新状态，通过日志文件记录变动，支持 Webhook 通知集成。

**导出格式适配** 支持将链接列表导出为 JSON、CSV、Markdown 表格等多种格式，方便导入第三方文档平台或知识管理工具。

**查询过滤器** 提供按条目 ID 范围、采集批次、文件扩展名（`.sHtML`）等条件的过滤查询，快速定位特定资源。

## 应用场景

技术文档编写与引用管理 技术作者在编写 API 文档、操作手册或技术博客时，可将本项目的链接作为参考资料的外部锚点，提升文档的可验证性与权威性。通过条目 ID 即可快速检索对应百科内容，无需重复手工搜索。

知识库自动化构建 企业或开源社区在搭建内部知识库时，可使用本项目作为数据源之一，将外部百科条目自动导入到知识图谱中，丰富概念间的关联关系。

链接健康度监控 运维团队或内容管理员可利用本项目的链接集合，配置定期可用性检查脚本，及时发现失效链接或内容迁移，保障引用链路的长期有效性。

教育资源共享 教育机构或在线学习平台可将本项目的条目分类作为课程参考资料索引，为学生提供结构化的扩展阅读清单。

## 快速开始

以下操作步骤适用于 Linux/macOS 环境，Windows 用户建议使用 WSL 或 Git Bash。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/wapwiki-resource-aggregator.git
cd wapwiki-resource-aggregator

# 安装依赖（Python 3.9+ 环境）
pip install -r requirements.txt

# 执行链接采集与索引构建（示例批次）
python run_collector.py --batch 38 --output ./data/batch_38.json
```

## 项目结构

```
wapwiki-resource-aggregator/
├── collector/                         # 核心采集模块
│   ├── __init__.py                    # 模块初始化与公开接口
│   ├── fetcher.py                     # HTTP 请求封装，含重试与代理支持
│   ├── parser.py                      # HTML 解析器，提取标题、摘要、分类
│   └── indexer.py                     # 构建条目 ID 与本地缓存映射
├── data/                              # 数据存储目录（按批次分置）
│   ├── batches/                       # 批次原始采集结果
│   │   └── batch_38.json              # 本批次输出样例
│   ├── indices/                       # 全局索引文件
│   │   └── master_index.db            # SQLite 主索引库
│   └── logs/                          # 运行日志
│       └── collector.log              # 采集器日志（按天滚动）
├── scripts/                           # 运维与辅助脚本
│   ├── run_collector.py               # 命令行入口，支持 --batch 参数
│   ├── export_formatter.py            # 导出为 JSON/CSV/Markdown
│   └── health_check.py                # 链接可用性巡检脚本
├── tests/                             # 单元测试与集成测试
│   ├── test_fetcher.py                # 模拟 HTTP 响应测试
│   ├── test_parser.py                 # 解析逻辑覆盖测试
│   └── fixtures/                      # 测试用静态 HTML 样本
├── docs/                              # 项目文档
│   ├── getting-started.md             # 快速入门
│   ├── collector-config.md            # 配置参数详解
│   ├── data-schema.md                 # JSON 数据结构规范
│   └── operations.md                  # 生产环境运维指南
├── requirements.txt                   # 生产环境依赖列表
├── requirements-dev.txt               # 开发环境额外依赖
├── Makefile                           # 常用任务快捷命令（test, lint, clean）
└── README.md                          # 本文件
```

## 贡献指南

欢迎开发者以任何形式参与本项目贡献。请遵循以下流程以确保代码质量和数据一致性。

提交 Issue 报告问题 在 GitHub Issues 中描述您遇到的问题或改进建议，请附带重现步骤、环境信息及相关日志片段。对于链接失效类问题，请提供具体的条目 ID 或完整 URL。

创建功能分支 从主分支 `main` 切出新分支，命名格式为 `feature/简述改动` 或 `fix/简述修复`。确保分支基于最新的主干代码。

## 常见问题

问：采集过程中遇到 HTTP 403 或 429 状态码应如何处理？

答：项目内置了指数退避重试机制（默认最大重试 3 次）。若持续遇到限制，请检查 `collector/config.py` 中的 `REQUEST_DELAY` 参数，适当调大请求间隔（建议 1.5 至 3 秒）。同时可配置代理池以分散来源 IP。若问题依旧，请提交 Issue 附带完整错误日志。

问：如何自定义采集的条目 ID 范围或过滤特定前缀？

答：在运行 `run_collector.py` 时可通过 `--id-range` 参数指定起始和结束 ID，例如 `--id-range 1000 5000`。若需按文件扩展名或 URL 模式过滤，可修改 `parser.py` 中的 `URL_PATTERN` 正则表达式。所有过滤条件均支持在配置文件中预定义。

问：采集结果中的 `.sHtML` 后缀与标准 `.html` 有何区别？

答：该后缀为源站点的实际文件命名规范，并非笔误。采集器按原样保存 URL 字符串，不进行规范化改写，以保证链接的可直接访问性。解析器在提取条目 ID 时使用正则匹配数字部分，不受后缀大小写影响。
