# LinkVault 聚合导航系统

LinkVault 是一个面向技术团队与内容运营者的轻量级外链资源聚合与导航系统。该项目专注于解决海量分散链接的集中管理、分类检索与快速访问问题，适用于需要维护大量外部参考资源的技术文档库、知识库或内部开发门户。LinkVault 不提供爬虫或自动采集功能，而是通过人工编排与结构化标注，将零散的 URL 转化为可维护、可扩展的知识导航体系。

## 功能概览

**层级化分类管理**：支持无限层级的目录树结构，允许管理员按技术领域、业务模块或使用频次对链接进行多级归类，每个链接可挂载至多个分类节点。

**全文元数据检索**：基于链接标题、描述、标签及自定义扩展字段构建倒排索引，支持模糊匹配与精确过滤，检索响应时间控制在 200 毫秒以内。

**访问状态健康检查**：内置异步任务调度器，定期对已收录链接进行 HTTP 头探测，自动标记失效链接并生成异常报告，支持手动触发重检。

**批量导入与导出**：提供 CSV 与 JSON 格式的批量链接导入接口，导出功能支持生成结构化数据包，便于迁移或备份。

**用户访问审计日志**：记录每次链接点击的来源 IP、时间戳与 Referer 信息，支持按时间范围聚合统计，辅助分析资源热度。

**权限分级控制**：基于角色的访问控制模型，区分管理员、编辑员与访客三种角色，编辑员仅可维护指定分类下的链接，访客只读。

**自定义标签体系**：允许用户创建全局标签库，每个链接可附加多个标签，支持标签组合筛选与快速定位。

**响应式前端界面**：基于 Bootstrap 5 构建的自适应布局，在桌面端与移动设备上均提供一致的浏览与检索体验。

## 应用场景

技术文档库的外部参考管理：技术写作团队在撰写 API 文档或架构设计说明时，需要引用大量外部规范、RFC 文档或社区讨论帖。LinkVault 可为每篇文档建立独立的链接清单，并自动检测链接有效性，避免文档中出现死链。

内部开发门户的导航聚合：企业研发中心通常维护多个内部系统（代码仓库、CI/CD 平台、监控仪表盘）。LinkVault 可作为统一的导航入口，将分散的管理后台地址按项目或团队分组，减少开发人员查找时间。

开源项目社区的贡献者引导：开源项目维护者可将 Issue 模板、贡献指南、编码规范、设计文档等外部资源集中收录于 LinkVault，新贡献者可通过导航树快速了解项目周边生态。

学术研究团队的文献参考库：科研人员在撰写论文或综述时，需管理大量预印本、数据集仓库与工具主页。LinkVault 的自定义标签与检索功能可辅助按研究方向、发表日期或作者对参考文献进行精细分类。

运维故障排查的知识索引：运维团队可将常见故障对应的日志查询地址、监控图表链接、历史事故报告等资源汇总至 LinkVault，配合健康检查功能，及时剔除已下架的监控面板。

## 快速开始

以下指令适用于 Linux / macOS 环境，需提前安装 Git、Node.js 18.x 与 pnpm。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core
pnpm install
pnpm run build
pnpm run start:dev
```

生产环境部署请使用 `pnpm run start:prod`，并配置 `.env.production` 中的数据库连接字符串与端口参数。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.17.0 或更高 | 运行时环境，建议使用 LTS 版本 |
| pnpm | 8.0.0 或更高 | 包管理器，用于依赖安装与工作区管理 |
| PostgreSQL | 14.0 或更高 | 主数据库，存储链接元数据、分类树与审计日志 |
| Redis | 6.2 或更高 | 缓存层，用于会话存储与健康检查任务队列 |
| Elasticsearch | 7.17 或更高 | 搜索引擎（可选），提供全文检索能力，未配置时降级为 SQL LIKE 模糊匹配 |
| Nginx | 1.22 或更高 | 反向代理（生产推荐），用于负载均衡与静态资源缓存 |

## 贡献指南

我们欢迎并鼓励社区提交补丁与新功能提案。请遵循以下流程以保证代码质量与协作效率。

1. 查阅 issue 列表与项目看板，确认当前开发优先级。若计划实现较大功能变更，请先创建讨论 issue 与维护者沟通设计思路，避免无效开发。

2. 从 `main` 分支拉取新的特性分支，分支命名遵循 `feat/功能简述` 或 `fix/问题简述` 格式。提交信息需符合 Conventional Commits 规范，包含类型、作用域与简短描述。

3. 编写或更新对应的单元测试与集成测试，确保新增代码的行覆盖率不低于 80%。对于涉及数据库模型变更的提交，必须同时提供 Prisma 迁移脚本。

4. 提交 Pull Request 前，执行 `pnpm run lint` 与 `pnpm run test` 确保本地检查全部通过。PR 描述中需清晰说明变更目的、影响范围以及手动测试步骤。

5. 维护者将在 7 个工作日内完成 Code Review，可能要求补充测试用例或调整接口设计。合并后即自动触发 CI 流水线构建并部署至预览环境。

## 常见问题

**Q: 健康检查功能是否会频繁请求外部链接，导致目标服务器产生额外负载？**

A: 系统默认将健康检查间隔设置为 24 小时，且每个周期内对同一域名仅发起一次 HEAD 请求，不下载响应体。管理员可通过环境变量 `HEALTH_CHECK_CRON` 自定义调度频率。对于内部敏感系统，建议关闭自动检查，改为手动触发。

**Q: 导入大量链接时，如何避免分类树性能退化？**

A: 分类树采用嵌套集模型存储，查询效率不受层级深度影响。但批量导入时建议每次提交不超过 500 条记录，并使用事务包裹。若需导入超过 10000 条链接，请通过命令行脚本 `pnpm run import:bulk --file=data.json` 异步执行，避免 HTTP 超时。

**Q: 是否支持链接去重检测？**

A: 系统在添加链接时会自动检查 `url` 字段的唯一性约束，若重复则返回 409 冲突状态码。对于 URL 编码差异（如 `%20` 与 `+`），系统在存储前会执行标准化处理，统一转为小写协议头并移除末尾斜杠。
