# LinkVault 资源聚合系统

LinkVault 是一个面向技术团队与内容运营人员的轻量级外链资源聚合与导航平台。该项目定位于将分散在各类百科、文档站点中的高价值技术文章、规范说明与操作指南进行集中收录、分类索引与快速检索，解决技术人员在多站点之间反复跳转查找资料的低效问题。项目本身不生产内容，而是通过结构化的资源编排与清晰的目录导航，帮助用户在一个入口处获取批次化、主题化的外链资源集合。

本批次为第 36 批资源入库，共计 180 个外链条目，全部来源于 wap.baike.zr54.cn 域下的技术百科类文章页面。这些链接覆盖了后端开发、前端工程、数据库运维、网络协议、系统安全等多个技术方向，经过本项目的索引编排后，可作为团队知识库的补充外链源，也可作为技术培训材料的外部参考附录。

## 功能概览

**批次化资源导入**：支持按批次（每批最多 200 条）将外部 URL 导入系统，自动识别 URL 结构并提取文章标识符，便于后续检索与归类。

**多维度分类索引**：每个资源条目可根据来源域名、URL 路径特征、文章编号段进行自动或手动的分类打标，支持按技术领域、难度层级、内容形式等维度筛选。

**原始链接透传**：系统对外展示的所有资源链接均保持用户提交时的原始 URL 格式，不进行协议升级、域名规范化或路径改写，确保目标站点的访问兼容性。

**状态监控与可用性检查**：定期对已入库的 URL 发起 HEAD 请求，检测目标页面是否可访问，并在管理后台标注失效链接，辅助运维人员清理或更新资源。

**检索与过滤接口**：提供基于文章编号、URL 关键字、入库批次号的检索能力，支持通过 API 或命令行工具批量导出指定分类下的完整链接列表。

**访问统计与热度排序**：记录每个外链在系统内的点击次数，支持按热度排序展示高频访问资源，帮助团队识别当前关注度较高的技术话题。

## 应用场景

技术团队内部知识库的扩展补充。团队在维护内部 Wiki 或 Confluence 时，可将 LinkVault 中收录的批次化外链作为「参考阅读」章节的素材来源，避免重复整理相同的外部链接。例如在撰写《微服务熔断策略实践》文档时，可直接从本系统检索相关百科文章并引用。

技术培训与新员工 onboarding 的阅读清单管理。培训组织者可按批次导出 LinkVault 中的链接列表，形成结构化的阅读任务单，学员点击链接即可直达目标文章，无需手动输入或查找。

技术博客与新闻通讯的链接素材池。内容编辑人员可定期从 LinkVault 的「高频访问」或「最新入库」列表中挑选外链，作为技术周报或博客「延伸阅读」板块的候选链接，提升内容产出效率。

个人开发者的书签与阅读列表管理。开发者可将 LinkVault 作为自己的公开书签管理工具，通过批次导入的方式批量添加零散收藏的百科文章，并借助分类索引功能快速回顾特定技术方向的内容。

## 快速开始

以下命令演示如何在本地环境部署 LinkVault 服务。

```bash
git clone https://github.com/your-org/linkvault.git
cd linkvault
pip install -r requirements.txt
python app.py --port 8080 --batch 36
```

上述操作完成后，服务将在本地 8080 端口启动，并自动加载第 36 批次的资源索引。访问 http://localhost:8080 即可查看本批次全部 180 个外链的导航页面。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Python | 3.8 及以上 | 核心运行环境，用于提供 Web 服务与 CLI 管理工具 |
| Flask | 2.0.x | Web 框架，用于渲染导航页面与提供 REST API |
| SQLite | 3.0 及以上 | 本地数据库，存储资源条目、分类标签与访问日志 |
| requests | 2.28.x | 用于发送 HEAD 请求以检测外链可用性 |
| click | 8.0.x | CLI 命令行工具框架，用于执行导入、导出、检测等管理操作 |
| gunicorn | 20.1.x | 生产环境 WSGI 服务器，用于多进程部署 |

## 项目结构

```
linkvault/
├── app/                               # 主应用模块
│   ├── __init__.py                    # Flask 应用工厂，注册路由与扩展
│   ├── routes/                        # 路由层，处理 HTTP 请求
│   │   ├── index.py                   # 首页与批次概览路由
│   │   ├── resource.py                # 单条资源详情与跳转计数路由
│   │   └── api.py                     # REST API 路由，提供 JSON 格式查询接口
│   ├── models/                        # 数据模型层
│   │   ├── batch.py                   # 批次模型，包含批次编号、导入时间、条目数量
│   │   ├── entry.py                   # 资源条目模型，包含 URL、文章编号、分类标签
│   │   └── click_log.py               # 点击日志模型，记录访问时间与来源 IP
│   └── services/                      # 业务逻辑层
│       ├── importer.py                # URL 列表导入服务，支持 CSV 与纯文本格式
│       ├── checker.py                 # 可用性检测服务，定时发送 HEAD 请求
│       └── classifier.py              # 自动分类服务，基于 URL 路径特征打标签
├── cli/                               # 命令行工具模块
│   ├── __init__.py                    # Click 命令组定义
│   ├── import_cmd.py                  # 导入新批次的 CLI 命令
│   ├── check_cmd.py                   # 手动触发可用性检测的 CLI 命令
│   └── export_cmd.py                  # 按条件导出链接列表的 CLI 命令
├── data/                              # 数据存储目录
│   ├── linkvault.db                   # SQLite 数据库文件，包含所有表与索引
│   └── seeds/                         # 初始数据种子目录
│       └── batch_36.csv               # 第 36 批次的原始 URL 列表文件
```

## 贡献指南

欢迎开发者为本项目提交改进。请遵循以下步骤参与贡献。

第一，在 GitHub 上 fork 本仓库至个人账户，然后克隆到本地开发环境。创建新的功能分支，分支名称应简要描述本次变更的内容，例如 feat/add-batch-import-json 或 fix/checker-timeout。

第二，编写代码或文档变更时，请遵守项目现有的代码风格。Python 代码应遵循 PEP 8 规范，所有新增函数需包含 docstring 说明其用途、参数与返回值。如需新增依赖，请在 requirements.txt 中明确版本号。

第三，在提交 pull request 之前，请在本地运行单元测试套件，确保所有现有测试用例通过。如果新增了功能模块，请同步编写对应的测试用例覆盖核心逻辑。

第四，提交 commit 时，请使用清晰的提交信息格式，建议采用 类型: 简短描述 的格式，例如 fix: 修复可用性检测服务对 HTTPS 链接的超时处理。提交信息应使用英文或中文均可，但需保持项目内统一。

第五，将变更推送到个人远程仓库后，在本仓库发起 pull request。PR 描述中应说明变更的目的、实现方式以及测试情况。项目维护者将在三个工作日内进行 review 并给出合并意见。

## 常见问题

问：导入批次时，系统是否会对 URL 进行去重处理？

答：是的。在导入过程中，系统会检查新增 URL 是否已存在于数据库中。如果发现相同 URL 已入库，则不会重复添加，并在导入日志中记录跳过项。去重依据为完整 URL 字符串，区分大小写与协议部分。

问：可用性检测服务多久运行一次，检测结果如何查看？

答：默认情况下，系统会通过系统定时任务（如 crontab）每 24 小时触发一次全量检测。检测结果会更新到每个资源条目的 last_checked 和 is_accessible 字段。管理员可通过 CLI 命令 linkvault check --report 生成当前失效链接列表，或在 Web 管理后台的「资源状态」页面查看。

问：是否支持自定义分类标签，还是只能使用系统预设的分类？

答：系统允许用户自定义分类标签。在导入时可以通过 CSV 文件中的 tag 列指定任意文本作为标签，系统会自动识别并创建新标签。现有资源条目也可以通过管理后台或 API 进行标签的增删改操作。标签系统不限制层级深度，但建议保持扁平化结构以便于检索。
