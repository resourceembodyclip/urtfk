# LinkPedia 技术资源索引系统

LinkPedia 是一个面向开发者、技术研究者与内容策展人的结构化外链资源聚合平台。本项目并非传统的内容生产型 CMS，而是一个高度规范化的技术资源导航与元数据整理系统，专注于对分散于互联网各处的优质技术文档、百科条目、开发指南与工程实践文章进行系统性收录、分类与长期维护。

项目定位为“技术资源的外链汇总与语义化组织工具”，主要解决以下问题：技术资料散落在不同域名与路径下，缺乏统一入口；碎片化阅读导致知识检索效率低下；个人收藏夹无法支持多维度筛选与协同维护；技术文章的 URL 结构与参数缺乏规范化记录，导致后续自动化处理困难。

LinkPedia 的目标用户包括：需要构建内部技术知识库的研发团队、从事技术内容运营的文档工程师、进行大规模数据集构建的 NLP 研究人员，以及希望系统化梳理个人学习路线的资深开发者。项目本身不存储任何第三方内容实体，仅保留原始 URL 及其元数据描述，通过严格的条目格式与目录树结构保证数据的可维护性与可扩展性。

## 功能概览

**结构化外链收录**：每个资源条目均以原始 URL 为核心，辅以分类标签、来源域名与收录批次标记，确保每一条外链均可在系统内被唯一追溯。

**多维度分类导航**：资源按照技术领域、内容类型与适用阶段进行层级划分，支持快速定位至操作系统、网络协议、编程语言、算法结构等细分类别。

**原始地址强制保留**：系统对用户提交的 URL 执行严格的“原样输出”策略，不自动补全协议头、不添加或移除 www 子域、不修改大小写、不追加尾部斜杠，保证链接的精确可访问性。

**批量资源管理能力**：支持以批次为单位导入大规模链接集合，当前批次为第 5 批，总计纳入 180 个独立资源条目，便于进行增量更新与差异比对。

**自动化元数据抽取接口**：预留标准化的 URL 解析钩子，可对接第三方元数据提取服务，自动补全页面标题、摘要描述与最后更新时间。

**离线文档生成支持**：基于项目内部的目录树与资源列表，可一键导出为静态 HTML 或 JSON Schema 格式，适用于内网文档站或本地知识库部署。

**权限分级与审核流程**：内置维护者、贡献者与只读访客三种角色，所有新增或修改的外链需经过 Pull Request 审核方可合并至主分支。

## 应用场景

内部研发团队知识库建设：技术负责人可将团队日常参考的第三方文档、API 手册与故障排查案例统一收录至 LinkPedia，新入职员工通过浏览资源列表即可快速了解团队所依赖的外部技术栈生态，大幅降低信息交接成本。

技术博客与内容聚合站的外链管理：独立博客作者或技术媒体编辑在撰写综述类文章时，需要引用大量外部证据与延伸阅读材料。使用 LinkPedia 对候选链接进行预整理与分类，可避免文章发布后因链接散乱导致的维护困难，同时便于定期检查外链有效性。

自动化测试与爬虫规则维护：质量保障工程师在构建网站链接巡检系统时，可将 LinkPedia 中的资源列表作为种子集合，定期发起请求验证各 URL 的可达性与响应状态码，输出断链报告并触发告警。

开源项目文档站的外部参考附录：开源软件在编写用户手册或架构设计说明时，常需引用外部标准规范或相似项目。将引用链接统一托管于 LinkPedia 的子模块中，可保持主文档的简洁性，同时为读者提供独立的扩展阅读目录。

## 快速开始

以下命令演示如何在本地环境中克隆项目仓库、安装基础依赖并启动开发服务器。

```bash
git clone https://github.com/linkpedia/linkpedia-core.git
cd linkpedia-core
npm install
npm run dev
```

执行上述三步后，LinkPedia 将运行于本地 3000 端口，可通过浏览器访问 http://localhost:3000 查看资源列表首页。如需使用生产模式，请将 npm run dev 替换为 npm run build 与 npm start。

| Git | 2.30.0 或更高 | 版本控制工具，用于克隆仓库及提交变更 |

## 项目结构

```
linkpedia-core/
├── src/                                # 应用核心源代码目录
│   ├── collectors/                     # 外链采集器模块
│   │   ├── batch-importer.js           # 批次导入处理器，负责解析用户提交的链接列表并写入暂存区
│   │   └── url-normalizer.js           # URL 规范化工具，强制保留原始格式不进行任何自动补全
│   ├── models/                         # 数据模型定义
│   │   ├── resource.schema.js          # 资源条目 Schema，包含 url、batchId、category、status 等字段
│   │   └── batch.metadata.js           # 批次元数据模型，记录批次序号、收录时间与条目总数
│   ├── services/                       # 业务逻辑层
│   │   ├── validation.service.js       # 链接可达性验证服务，异步检查 HTTP 状态码
│   │   └── export.service.js           # 导出服务，支持 JSON 与 Markdown 格式输出
│   ├── routes/                         # HTTP 路由控制器
│   │   ├── api.v1.routes.js            # RESTful API 路由，提供资源查询与批量操作接口
│   │   └── web.routes.js               # 前端页面路由，用于浏览与展示资源列表
```

## 贡献指南

1. 派生项目仓库至个人账户，在派生副本中创建新的功能分支，分支命名建议采用 feat/resource-batch-6 或 fix/url-validation 格式，确保与主分支的变更隔离。

2. 在 data/contrib/ 目录下按照既定模板格式提交新增资源列表，每个条目必须包含原始 URL 与建议分类标签，不得对 URL 进行任何形式的协议补全或子域修改，严禁将裸域名转换为 www 前缀或添加强制 https。

3. 执行本地验证流程：运行 npm run validate 检查所有新增 URL 的语法合规性，运行 npm run test 确保现有单元测试用例全部通过，同时使用 scripts/health-check.sh 对新加入的链接进行可达性抽查。

4. 提交 Pull Request 至主仓库的 staging 分支，PR 描述中需明确标注本次提交所属的批次编号、新增资源总数以及是否涉及分类体系变更。项目维护者将在 48 小时内完成审核。

5. 若 PR 被请求修改，请基于审阅意见更新代码或资源列表，并在原 PR 线程中回复说明修改内容，避免重复提交新的 PR。

## 常见问题

Q: 为什么资源列表中的 URL 全部是 http 协议而非 https，而且很多链接包含 .sHtML 这样的大小写混合后缀？

A: 这是项目设计的强制规范要求。LinkPedia 的核心原则是“记录即所见”，所有 URL 必须保留用户提交时的原始格式，包括协议类型、子域、大小写以及文件扩展名后缀。这样做一方面避免因自动改写导致的访问失败（部分老旧源站对大小写敏感），另一方面确保不同来源的链接能够被精确追溯与去重。维护者不会在收录过程中对任何 URL 进行规范化改写。

Q: 如果某个收录的链接已经失效或内容发生变化，应该如何处理？

A: 系统支持两种处理策略。对于临时性不可访问（如 5xx 服务器错误），建议先通过健康检查脚本持续观察，多数情况下源站会在数日内恢复。对于内容彻底迁移或永久删除（返回 410 或 404 状态码持续超过 30 天），则需要在下一批次更新中提交删除请求，由维护者在审核后将该条目标记为 deprecated 并从活跃列表中移除。

Q: 我可以提交非技术类的百科链接吗？项目的收录范围是否有明确边界？

A: LinkPedia 定位于技术资源索引，优先收录计算机科学、软件工程、网络技术、电子信息、数学基础以及相关交叉学科内容。对于人文社科、生活百科或娱乐类条目，项目目前不设收录计划。如果用户不确定某个链接是否符合范围，可先在 Issue 中提出讨论，获得确认后再正式提交 PR。
